## Root cause of the "gained/lost/never" pattern in `final_run_1`

Traced directly against `/home/quinten/fetch` source (not the study harness) after `final_run_1`'s per-case results showed 24 cases gaining the expected exact label after the disclosed fact, 31 losing it, and 145 matched cases never landing on it either before or after. Two structural bugs in FETCH's core `classify()` path, both firing specifically on the second (`followup_answers`-bearing) call:

1. **GPT-5-family models never receive the follow-up answer on the normal code path.** `app/providers/openai.py:177-179` builds the Responses API `input` as `f"SYSTEM:\n{prompt}\n\nUSER:\n{problem_description}"` — using only the original opening query. The `followup_answers` parameter (line 104) is never referenced in the `GPT_5_FAMILY_MODELS` branch (lines 155-220); it's only consumed by `build_messages()` inside `except TypeError`/`except AttributeError` fallback handlers (lines 254-256, 270-272), which fire only if the Responses API call itself throws. In the normal successful case — the overwhelming majority — call 2 sends the byte-identical prompt as call 1. Since this study's classifier is `gpt-5`, **the disclosed fact was never actually seen by the model on either official run.**
2. **The provider-pool filter for refinement calls is over-broad.** `app/services/classification_service.py:1250-1261` filters `current_providers` to `isinstance(p, LLMClassifierProvider)` when `followup_answers` is present, with a comment saying this excludes "non-LLM providers like spot/keyword." Verified directly: `KeywordClassifierProvider` and `SpotProvider` are correctly excluded, but `GeminiProvider` and `MistralProvider` are *also* not subclasses of `LLMClassifierProvider` (only `OpenAIProvider` is) and get excluded too, despite being real LLMs. For this study's `["gpt-5", "keyword"]` config, this collapses the vote pool from 2 providers (weight 0.9 + 0.5) to 1 (weight 0.9 alone) on call 2 — any label that call 1 only surfaced because keyword co-voted for it has no path to survive into call 2.

Together: call 2 has exactly one voting provider, and that provider is re-answering the identical prompt it already answered in call 1 (GPT-5-family calls also never set `temperature`, only `reasoning_effort`, so repeat calls to the identical prompt are not even deterministic). The "gained/lost" split is best explained as GPT-5 resampling noise between two identical calls, not answer-driven reasoning — which is also why a second full run would not be informative: this is a pipeline defect present on every call, not something a larger sample size would average out.

Full citations and the fix recommendations discussed with the study owner are in `RESULTS.md`.
