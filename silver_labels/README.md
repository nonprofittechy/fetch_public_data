# Silver-label creation

This directory contains the reproducible artifacts for the AI-generated labels
for `redaction_reviewed_v5_clean.xlsx`.

## Final artifact

[`04_review/redaction_reviewed_v5_clean_ai_silver_reviewed.xlsx`](04_review/redaction_reviewed_v5_clean_ai_silver_reviewed.xlsx)
contains the original three reviewed columns plus, for every input row:

- up to three ranked category/subcategory matches from GPT-5.2, Gemini 3.1 Pro Preview, and DeepSeek v4;
- each model's concise justification and confidence;
- `review_category`, `review_subcategory`, and `review_justification`;
- `review_basis` and `reviewed_by_ai`.

All 431 rows are highlighted pale yellow because they received an AI label. The
three model-specific workbooks in `01_gpt52/`, `02_gemini31_pro/`, and
`03_deepseek_v4/` provide the independent outputs before review.

The complete row-by-row audit is in
[`04_review/final_review.json`](04_review/final_review.json). It includes the
input text, all ranked model candidates, the reviewed choice, and the review
basis for every row.

## Canonical taxonomy and validation

Every model prompt includes the full contents of
`../app/data/taxonomy_detailed_descriptions.csv`, not the simpler label-only
taxonomy. The file has 209 detailed entries. The one capitalization typo
`General litigation` was normalized to `General Litigation`; no descriptions or
subcategory names were changed.

Each returned category/subcategory pair was checked against the exact pair set
from that detailed taxonomy. Invalid or invented pairs caused the batch to be
retried rather than being silently accepted. Keyword and SPOT classification
were not called.

## Independent model passes

Each pass used the same prompt builder, same taxonomy snapshot, same 20-row
batches, same JSON schema, and the same validation logic. Only the model route
changed:

| Folder | Provider route | Model | Credential variable |
|---|---|---|---|
| `01_gpt52/` | Azure OpenAI-compatible `/chat/completions` | `gpt-5.2` | `OPENAI_API_KEY`, `OPENAI_BASE_URL` |
| `02_gemini31_pro/` | Gemini `v1beta ...:generateContent` | `gemini-3.1-pro-preview` | `GEMINI_API_KEY` |
| `03_deepseek_v4/` | Azure OpenAI-compatible `/chat/completions` | `deepseek-v4` | `OPENAI_API_KEY`, `OPENAI_BASE_URL` |

Credentials are read from the process environment or `../.env` at runtime and
are not written to any artifact. `run_metadata.json` records the model,
provider, source-file hashes, taxonomy snapshot hash, and prompt builder. Each
folder's `prompts/batch_*.txt` is the exact rendered prompt sent for that batch.
The alternative-model folders also contain the raw provider JSON responses in
`responses/`; `checkpoint.json` contains the normalized, taxonomy-valid labels.

## Exact prompt

The prompt is generated by `build_prompt()` in
[`../create_silver_labels.py`](../create_silver_labels.py). Its structure is:

```text
You are the final LLM classifier in a legal-help routing pipeline.

Classify each user problem below using ONLY the canonical taxonomy entries and
their detailed descriptions. Do not use a keyword classifier, SPOT, or the
existing expected labels as a shortcut. Reason about the user's role, the legal
issue, and the distinction between neighboring subcategories.

Return up to three possible matches for each row, in descending order of fit.
Use exact category and subcategory strings from the taxonomy. Do not invent a
label. Usually return one match; return two or three only when the text really
leaves a meaningful ambiguity. Each justification must be concise (one or two
sentences), identify the facts that support the match, and explain the most
important distinction when a lower-ranked match is included. Always provide a
best effort match, even when the description is sparse.

Return ONLY valid JSON in this shape:
{
  "classifications": [
    {
      "row_id": 1,
      "matches": [
        {"category": "...", "subcategory": "...", "justification": "...", "confidence": "high|medium|low"}
      ]
    }
  ]
}

CANONICAL DETAILED TAXONOMY:
<all 209 taxonomy rows, each as category | subcategory | enriched description>

ROWS TO CLASSIFY:
<up to 20 rows as ROW <spreadsheet row number>: <problem description>>
```

The actual `<...>` sections are preserved in each batch prompt file, so the
exact text—not just this summary—can be audited or replayed.

## In-context review pass

The review was performed by `build_reviewed_silver.py`, with no model/API call
and no use of the supplied credentials. The review procedure was:

1. Compare each model's rank-1 category/subcategory pair.
2. Keep unanimous three-model choices as three-model consensus.
3. Keep two-model choices as two-model consensus unless the row was one of the
   explicitly ambiguous cases reviewed against the detailed descriptions.
4. For ambiguous rows, read the problem text, the competing justifications,
   and the relevant detailed taxonomy descriptions; select the most specific
   supported match and record the rationale in `final_review.json`.
5. Revalidate every final reviewed pair against the detailed taxonomy.

There were 353 unanimous rank-1 rows, 72 two-model rank-1 agreements, and 6
three-way rank-1 disagreements before review. The review record marks 77 rows
as `in-context manual review`; the remaining rows retain a documented model
consensus basis.

## Reproduction commands

From this directory's parent repository:

```bash
python create_silver_labels.py \
  --provider azure --model gpt-5.2 \
  --checkpoint silver_labels/01_gpt52/checkpoint.json \
  --run-dir silver_labels/01_gpt52 \
  --output silver_labels/01_gpt52/redaction_reviewed_v5_clean_gpt52.xlsx

python create_silver_labels.py \
  --provider gemini --model gemini-3.1-pro-preview \
  --checkpoint silver_labels/02_gemini31_pro/checkpoint.json \
  --run-dir silver_labels/02_gemini31_pro \
  --output silver_labels/02_gemini31_pro/redaction_reviewed_v5_clean_gemini31_pro.xlsx

python create_silver_labels.py \
  --provider azure --model deepseek-v4 \
  --checkpoint silver_labels/03_deepseek_v4/checkpoint.json \
  --run-dir silver_labels/03_deepseek_v4 \
  --output silver_labels/03_deepseek_v4/redaction_reviewed_v5_clean_deepseek_v4.xlsx

python build_reviewed_silver.py
```

The checkpoint makes interrupted model passes resumable. Delete or rename a
pass's checkpoint to intentionally regenerate that pass.

## Human-label cross-check

[`05_human_label_review/`](05_human_label_review/) records a separate check of
the nine cases explicitly discussed as error examples in
`interim_not_to_commit/thats_so_fetch.pdf`, Table 3 (PDF page 199). Four human
labels were corrected in the derived checked workbook:

- row 149: Consumer Law → Criminal Law / Post Conviction/ Appeals;
- row 188: Criminal Law → Consumer Law / Small Claims Advice;
- row 220: Debtor/Creditor → Administrative Law / Unemployment;
- row 339: Labor & Employment → Administrative Law / Professional Licensing.

The original human columns in `redaction_reviewed_v5_clean.xlsx` are unchanged.

For the next round of manual review, use
[`06_human_review_workspace/`](06_human_review_workspace/). It includes all
ranked model candidates, the internal label, blank blue human-entry fields, the
four-pass agreement fields, and the 78-row disagreement extract.

For the multi-label question, use
[`07_multilabel_audit/`](07_multilabel_audit/). That stage found 156 of 431 rows
where at least two distinct exact taxonomy pairs were proposed by at least two
of the three independent models. This is a deliberately broad evidence queue,
not a claim that all 156 rows definitely need multiple labels. Its workbook
shows up to three repeated candidates, the supporting models and justifications,
and three blank human label slots. The stage README defines the evidence rule
and the necessary distinction between a separate second issue and a weaker
alternative framing.

The follow-up Azure GPT-5.2 two-label audit is in
[`08_gpt52_two_label_audit/`](08_gpt52_two_label_audit/). It independently
audited the final AI primary label while allowing at most two labels per row.
It found 90 two-label rows, 81 primary-label change candidates, and 12
uncertain primary assessments. These are fresh single-model audit signals for
human review, not automatic corrections.
